## About

DevContractShield is a **decentralized escrow and adjudication platform for freelance software delivery contracts**. It eliminates subjective disputes between clients and developers by enforcing objective, technically-verifiable acceptance criteria through an **on-chain AI adjudicator** powered by [GenLayer Intelligent Contracts](https://docs.genlayer.com/).

### The Problem

Freelance software projects frequently end in disputes: *"the code doesn't work"* vs *"it meets the requirements."* Traditional escrow services rely on human mediators who lack the technical context to evaluate code quality objectively.

### The Solution

DevContractShield automates the entire acceptance workflow:

1. **Client creates a contract** with clear acceptance rules (e.g., "all tests must pass and linter must be clean").
2. **Developer submits their GitHub repository** when work is complete.
3. **The platform clones the repo and runs automated tests** against the client's acceptance criteria.
4. **Test evidence is submitted to a GenLayer Intelligent Contract**, where an LLM-based on-chain adjudicator evaluates the results.
5. **Funds are released or refunded** automatically based on the verdict — no human mediator needed.

## How It Works

```
┌──────────┐     ┌──────────────┐     ┌────────────────┐     ┌──────────────────┐
│  Client   │────▶│  Create      │────▶│  Developer     │────▶│  Platform runs   │
│  deposits │     │  contract +  │     │  submits repo  │     │  automated tests │
│  GEN      │     │  acceptance  │     │  URL           │     │  (node --test)   │
└──────────┘     │  rules       │     └────────────────┘     └────────┬─────────┘
                 └──────────────┘                                     │
                                                                      ▼
┌──────────┐     ┌──────────────┐     ┌────────────────────────────────────────┐
│  Payment  │◀───│  Verdict:    │◀────│  GenLayer Intelligent Contract         │
│  released │     │  ACCEPTED or │     │  LLM adjudicator evaluates evidence   │
│  or       │     │  REJECTED    │     │  via gl.eq_principle (dual consensus) │
│  refunded │     └──────────────┘     └────────────────────────────────────────┘
└──────────┘
```

### Three-Layer Evidence Model

The adjudicator evaluates submissions using a structured evidence bundle:

| Layer | What it checks | Weight |
|-------|---------------|--------|
| **Layer A** — Client Tests | Client-defined test suite pass rate | Must equal `1.0` (100%) |
| **Layer B** — AI-Suggested Tests | Informative quality tests generated by AI | Advisory only, excluded from verdict |
| **Layer C** — Linter & Runner | Linter status + runner exit code | Must equal `"pass"` |

The acceptance rule is enforced on-chain: `layer_a_pass_rate == 1.0 AND layer_c_status == pass`.

## Getting Started

### 1. Clone the repository

```bash
git clone git@github.com:RoodrigoReyes/DevContractShield.git
cd DevContractShield
```

### 2. Set up the backend

```bash
cd backend
cp .env.example .env
```

Fill in your `.env` with the required values:

```env
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_ANON_KEY=your-anon-key
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key
DATABASE_URL=postgresql://postgres:password@host:5432/postgres
JWT_SECRET=your-secret-key
OPENAI_API_KEY=sk-...
PORT=3000
NODE_ENV=development
```

Then run the database setup and start the server:

```bash
npm install
npm run start:dev
```

The API server will be available at `http://localhost:3000`.

### 3. Set up the frontend

```bash
cd frontend
cp .env.example .env.local
```

Configure the backend URL:

```env
NEXT_PUBLIC_API_BASE_URL=http://localhost:3000
API_BASE_URL=http://localhost:3000
```

Then install and run:

```bash
npm install
npm run dev
```

The frontend will be available at `http://localhost:3001` (or the next available port).

### 4. Deploy the Intelligent Contract (optional)

To deploy the GenLayer contract for on-chain adjudication:

```bash
# From the project root
genlayer network          # Select your network (studionet, localnet, testnet-*)
genlayer deploy           # Deploy contracts/dev_contract_shield.py
```

Or use the interactive Jupyter notebook at `contracts/sandbox_adjudication.ipynb` for a step-by-step walkthrough.

## The Intelligent Contract

The `dev_contract_shield.py` contract is deployed on GenLayer and manages the full escrow lifecycle:

### State Machine

```
Created → Funded → Submitted → UnderReview → Approved  → Finalized
                                            → Rejected → Finalized
                                            → Appealed (dispute)
```

### Key Operations

| Method | Who | Description |
|--------|-----|-------------|
| `fund_contract()` | Client | Deposits GEN tokens into escrow |
| `set_test_pack_hash(hash)` | Platform | Registers the official test pack hash on-chain |
| `set_submission_hash(code_hash)` | Developer | Registers the submitted code hash |
| `submit_evidence(evidence_summary)` | Platform | Submits test results; LLM adjudicates on-chain |
| `release_payment()` | Platform | Releases funds to developer (if approved) |
| `refund_client()` | Platform | Returns funds to client (if rejected) |
| `open_dispute(reason)` | Either party | Opens a dispute for manual review |

### On-Chain Adjudication

The contract uses `gl.nondet.exec_prompt` to have an LLM evaluate the evidence bundle, and `gl.eq_principle.prompt_comparative` to ensure two independent LLM instances reach the same verdict — providing **dual-consensus adjudication** that is verifiable and tamper-resistant.

## Dashboard Features

### For Clients

- **AI-Assisted Contract Creation** — An integrated chat assistant (powered by GPT-4o-mini) helps define clear, measurable acceptance requirements through a conversational interface.
- **Contract Management** — Track all contracts with real-time status updates across the full lifecycle.
- **Wallet Integration** — Fund contracts with GEN tokens directly from the dashboard.

### For Developers

- **Repository Submission** — Submit a GitHub repository URL when work is complete.
- **Live Test Execution** — Watch automated tests run in real-time via SSE-streamed terminal logs.
- **Coverage Reports** — View test pass rates and coverage metrics for each submission.

### Shared

- **Role-Based Routing** — Clients and developers see tailored views based on their role.
- **Dark/Light Theme** — Full theme support with system preference detection.
- **Contract Detail View** — Complete contract information including GenLayer transaction status, adjudication results, and evidence history.

## Database Schema

The platform uses four main tables in Supabase (PostgreSQL):

- **`users`** — User accounts with roles (client/developer)
- **`contracts`** — Software delivery contracts with acceptance rules, amounts, deadlines, and coverage metrics
- **`adjudications`** — Verdict records with layer-by-layer evidence (JSONB), reasons, and applied rules
- **`contract_system_statuses`** / **`genlayer_transaction_statuses`** — Lookup tables for state machine and GenLayer consensus phases
